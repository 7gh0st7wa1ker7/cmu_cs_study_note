# Debugging

本节主要是告诉你：**如何定位*Bug***。高级程序员的能力差异，不在于写代码，而在于 **快速定位问题的能力**。

本节中介绍了使用 `print()/locals()/input()` 等方法进行调试，进而发现 *Bug*。但是随着技术的发展，很多 *IDE* 工具都提供了更好的断点调试工具，工具一直在变，但是整个 *Debugging* 的思路还是一致的，所以我们主要是总结本节想要传递的排查思路。

# 排查思路

## Debug 的本质

程序卡死或者输出不符合预期，真正的问题是：**哪个变量没有按照预期变化**。
调试其实就是 **观察状态的变化**，状态指的是 **程序中的变量，比如定义的方法函数以及函数中的变量**，我们通过断点的方式，一步一步跟踪*状态*的变化过程，从而定位到再哪一步操作的时候出现了误差，导致问题的出现。

所以*Debug*的核心：
```text
观察状态
↓
发现异常状态
↓
定位代码
↓
修复
```

## 实际项目建议

### 不信任一切未知的输入

不信任一切未知的输入，哪怕是本项目方法函数的输入输出，除非你确定对方输出的结果是经过严格校验。

比如 *API*返回异常，数据库查询结果为空，类型错误等等，多一些防御性的校验会让你的程序更加健壮。

### 善用 logger

一个正在运行的项目是没有断点给你打的，所以在一些关键的输入输出逻辑中，使用 *logger* 将一些关键的信息提前抛出来，能帮助我们更快速的定位问题。

实际的项目中排错流程：
```text
发现错误日志 -> 定位错误方法函数 -> 复现 Bug -> 观察状态 -> 找到不变量 -> 修复 -> 测试 -> 增加测试方法
```

# 总结

**不要猜测程序在做什么，而是让程序告诉你它在做什么。**工具都是手段，真正重要的是：**Debug = 观察程序状态变化，而不是凭感觉修改代码。**
