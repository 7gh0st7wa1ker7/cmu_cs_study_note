# 代码风格

本节核心：**代码首先是给人读的，其次才是给机器执行的。**机器不在乎 还是 ，机器的执行速度一样。
但是程序员会因此：
- 减少 *debug*
- 少出*bug*
- 更容易维护

所以在真实项目里：**好的 Style 不是美观问题，而是工程效率问题。**

很多优秀项目的代码之所以看起来“简单”，并不是因为功能简单，而是因为程序员在这些方面花了大量精力：
- 拆分函数；
- 命名
- 消除重复代码
- 去掉 Magic number
- 减少全局状态
这其实比学习 *Python* 语法本身更重要。

## 功能拆分

在大型项目中，一个功能模块可能涉及多个执行动作，如果将这些执行动作都放在一个函数方法中，这种函数是无法维护的。比如：
```python
# 订单处理逻辑
def processOrder():
    # 参数验证
    validateOrder...
    # 数据库查询
    deductInventory...
    # 发送通知
    sendEmail...
    # 日志落库
    writeLog...
```

但是如果将这些动作单独拿出来，写成一个更小的功能函数，那么整个代码逻辑就会简单很多：

```python
def processOrder():
    # 参数验证
    validateOrder()
    # 数据库查询
    deductInventory()
    # 发送通知
    sendEmail()
    # 日志落库
    writeLog()
    return success
```

每个函数大约保持在：20～30 行左右。

这就是 **Top-down Design**（顶层描述流程）。

### Helper Function

除了功能模块的拆分，在实际项目开发过程中，会有很多重复的代码逻辑，比如数据解析和数据库查询等代码逻辑，这时候，我们在设计或编写代码过程中需要将这种重复的代码块提取出来，合并成通用的方法函数，这样就能大大减少项目中的 **重复代码逻辑**。

### Test Functions

这是最容易被低估的东西。实际上：**测试代码的重要性可能高于业务代码。**这样的话，以后修改，自动验证，可以在更前的阶段发现一些bug。

#### 边界测试

很多 `bug` 出在边缘值，例如：`isPrime(2)\isPrime(0)\isPrime(-3)` 。

### Efficiency

在设计架构和方法函数的时候，需要考虑 *效率* ，也就是函数计算的复杂度 *O(n)*。复杂度不是指*程序快不快*，而是研究 **当数据扩大 100 倍，程序会慢多少倍**，可以说
> 算法 = 数据结构 + 时间复杂度
而*时间复杂度* 的本质就是研究“工作量随着输入规模如何增长”。

#### 复杂度的计算

时间复杂度（Order of n）：单位：`O(n)`，表达的意思是数据扩大 10 倍，运行时间扩大 10 倍。

- 一层循环 `for i in range(n) -> O(n)`;
- 两层循环 `for i in range(n): for j in range(n): -> O(n**2)`
- 两个连续的循环 `for i in range(n): ... && for j in range(n) -> O(2n)`
- 一半规模 `while n > 1: n //= 2 -> O(log n)` (二分查询)

所以后续在使用循环方法的时候，需要注意能减少嵌套层，或者减少需要过滤的数据量，来降低计算量。

## 格式

格式对于机器来说是没区别的，唯一影响的就是人的阅读，进而影响后续维护和*debug*的效率。

### 注释

注释解释的是为什么，而不是做什么。比如：

```python
# 遍历用户 ，❌ 通过代码就能看出来
# 过滤已禁用的用户，防止发送失败报错，✅ 方便了reviewer和审计人员
for user in users:
```

如果对代码逻辑进行了修改，需要确认当前的注释是否还符合，是否需要修改，不然错误的注释会给reviewer和审计人员/维护人员带来额外的麻烦，也会让人觉得你不专业。

### 变量命名

不要用 `a,b,c...` 作为变量名，不利于维护。变量名称不宜过长，尽量控制在3～5个单词内。

#### Bool 变量

建议使用 `is/can/has` 后跟具体状态名称，如 `isActive/hasPermission/canWrite`。

#### int/float 变量

建议使用 *Count/Price* 前面接对象名称，如 `userCount/totalPrice`。

#### 集合变量

建议直接使用对象名称的复数表示，如 `users/servers/orders`。

#### 当前对象

建议使用 `current` + 对象。

#### 临时变量

在循环中，可以使用 `i,j,k...`

#### 全局变量

对于一些约定好的常量，可以使用全大写在项目中定义好，比如 `MAX_RETRY/DEFAULT_TEMPERATURE/REQUEST_TIMEOUT_SECONDS`。

不要在方法函数中对全局变量进行修改，不然这将是一个定时炸弹，你永远不知道它会在什么时候爆炸。

最后，不要使用内置函数名来进行命名，这会导致未知错误。
